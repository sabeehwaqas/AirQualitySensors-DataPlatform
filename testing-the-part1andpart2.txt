### P1.5 — Force a threshold breach (if `ingest_ms` isn't naturally breaching)

The threshold is `MAX_AVG_INGEST_MS=50` in docker-compose. To guarantee an alert without needing resource limits, temporarily lower it:

In `docker-compose.yaml`, change:
```yaml
MAX_AVG_INGEST_MS: "5"   # temporarily lower to 5ms to force alerts
```
Then `docker compose up -d` (no rebuild needed for env var change). After capturing alerts, set it back to `"50"`.

### P2.4c — Local vs GCS performance comparison

You need two runs: one with GCS disabled, one with GCS enabled. The easiest way is to temporarily blank out `GCS_BUCKET` in docker-compose.

**Run 1 — Local only:**
```bash
# In docker-compose.yaml, comment out GCS_BUCKET:
#   GCS_BUCKET: "airqualitybdpasg"

docker compose up -d   # no rebuild needed, just env var change

# Trigger runs and note the timings
curl -X POST http://localhost:8003/tenants/tenantA/run
curl -X POST http://localhost:8003/tenants/tenantB/run

# Capture the logs
curl http://localhost:8003/tenants/tenantA/logs
curl http://localhost:8003/tenants/tenantB/logs
```

Note `extract_sec` and `cache_mode=local`.

**Run 2 — Local + GCS:**
```bash
# Restore GCS_BUCKET in docker-compose.yaml
docker compose up -d

curl -X POST http://localhost:8003/tenants/tenantA/run
curl -X POST http://localhost:8003/tenants/tenantB/run
curl http://localhost:8003/tenants/tenantA/logs
```

Note `extract_sec` and `cache_mode=local+gcs`. The GCS run will be ~0.3–1s slower due to the network upload. You already have a GCS run from your screenshot (`extract_sec=0.408` with `gcs_upload_sec=0.403`) so you mostly just need the local-only run for comparison.
